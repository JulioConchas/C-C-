Architecture

How Titanium Works

  • You write Javascript
  • Titanium maps your JavaScript code to native components
  • Communication is two-way across the bridge

The key points to understand are:
  • You write your mobile app in JavaScript
  • When you create something, like a button, in JavaScript, Titanium creates a native equivalent for you
  • When you modify the button (set it's title or color), Titanium maps that change to native land
  • When the user clicks the button, your JavaScript code can react

Titanium Key Points:

  • You write in JavaScript
  • There's a platform-specific bridge and set of native components
    (so your source code can run on multiple platforms)
  • Your JavaScript is still there and running on the user's device
  • Your app is compiled (so your code is secure)

Cross-Platform in Titanium

  • Cross-platform !== 'Write Once, Run Everywhere'
  • Titanium is 'Write Once, Adapt Everywhere'
  • Goal is to build 'Best of Breed' apps
  • Non-visual code up to 100% portable
  • Accept and embrace platform differences

Titanium Provides APIs For:

  • UI components — add native buttons, pickers, tables and more to your apps
  • Networking — HTTP/HTTPS, sockets, Bonjour
  • Local storage — file, database, app properties
  • Camera, GPS, accelerometer
  • Social APIs — Facebook, YQL, more
  • Extensible & modular to add on custom functionality

Development Process

App Dev Process

  1. Write your source JavaScript
  2. Build for simulator/emulator/test device
  3. Debug/test
  4. Publish

Build Process

  1. Titanium Node.js scripts analyze your JavaScript and interact with native tools
  2. Your code is packaged into native project
  3. Platform-specific interpreter included
    • iOS: JavaScriptCore
    • Android: V8
  4. Native project is compiled using platform-specific tools
  5. If building to device, project is code-signed using native tools
Key point is that your JavaScript code is not accessible to end users. In production, the code is minified, obfuscated, and encrypted, then compiled within a native project. Even with this, we're not cross-compiling. Your JavaScript is still evaluated at runtime on the native platforms.

Debugging/Testing Overview

  • Syntax checking in Studio
  • Logging messages
  • Breakpoint debugging in simulator/emulator and device
  • Native tools for memory inspection
  • Appcelerator Test for automated functional testing
We'll cover the details at a later time. But here are some of the ways you can debug and test your apps. Studio provides syntax checking and will warn you before building a project with a simple syntax error. You can output logging messages to the console, which is accessible in the simulator/emulator and on device. You can use breakpoint debugging in the simulated environment and on device. And you can use native tools, such as Instruments, with varying levels of capability to inspect memory and performance issues.

Publishing Overview

  • Your code is minified and further compiled than with development builds
  • Images are optimized with a pngCrush style optimizer
  • App is signed with your publication certificates
  • Stored on the file system (Android) or synced to Xcode Organizer (iOS) for native publishing process
    (AdHoc & Enterprise publishing a bit different)
Again, we'll cover the details later. But Studio and the Titanium Mobile build scripts provide extra steps for the publishing process. Your code is optimized, as are graphics. Your app is signed with your publication certificates and stored as appropriate for your target platform. Then, you can continue with the vendor's tools to upload and publish your app. We'll cover the subtleties of App Store, AdHoc, and Enterprise publishing in the publishing lesson later in the course.

Jira Tickets

jira.appcelerator.org

To report a bug, or to determine if a behavior you're seeing is a bug, you can visit our Jira site at jira.appcelerator.org. You log on using your Developer account. New tickets go into the Community pool, where they are triaged by Appcelerator staff. Make sure to provide all the details you can. And critically, provide a code sample that demonstrates the issue you're reporting. Make it concise and minimal with nothing extra that would confuse or distract the reviewers. Another tip: tickets with more watchers get prioritized. So watch the tickets you want resolved.
