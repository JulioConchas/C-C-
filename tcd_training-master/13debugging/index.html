Setting Breakpoints

  • Margin-click method:
    • Double-click margin
    • Do debug build
  • Debugger statement
    • Add debugger statement
    • Do debug build
You can use either method to introduce a breakpoint into your app. The margin-click method will not work with TiSDK 3.0 through 3.0.2, though will work with SDK 3.1+ apps.

Try it

Debugging AlloyHunter
  1. Add a breakpoint to the Fugitives.js controller as the first statement in the table event listener.
  2. Do a debug build of the app to the simulator/emulator.
  3. Click a row in the fugitives table.

Students should do this, and you should demo or talk through the steps as they go.

Debugging on Device

Appcelerator Platform Tools

Appcelerator Performance Management

Performance Management lets you monitor exceptions and crashes in your apps. Stats are sent live to the Dashboard. You can set breadcrumbs (text strings) that are visible on the Dashboard to help you note what conditions or actions led to the error condition.

Appcelerator Test

Test enables you to create automated tests for your apps, run on device or simulator/emulator.

Code Analysis

  • Code Analyzer
  • Code Profiler
  • Currently, only "traditional" apps, not Alloy
Code Analyzer looks for code that might cause problems -- accessing APIs not supported on a given platform, runtime-error detection, deprecated Titanium API warnings, etc. Code Profiler provides performance data to help you identify bottlenecks in your app.

Tips and Notes

Android Tips

  • Much more sensitive to data types (text.value=var will throw error if var != 'string')
  • Less than forgiving than iOS with syntax errors
  • Use Monitor/ddms to view logs. Studio sometimes doesn't show all
  • Check out adb features: remote shell, push/pull files, launch sqlite to access an app's database
Android tends to be much more sensitive to data types. It will often throw an cryptic error if you pass the wrong data type. Try passing a boolean to textfield.autocapitalize or setting textfield.value = 123 to see such crashes. iOS in comparison will often keep happily chugging along. Use Monitor (or ddms) to view the complete logs. And check the Android developer docs for info on using adb to access the emulator session (or rooted device).

Simulator vs. Emulator

iOS SimulatorAndroid Emulator
Description: App runs as OS X process App runs in real OS inside virtual computer
Good: Fast, no code signing steps, use native tools (Console, Instruments) for debugging Emulate many types of hardware, screen sizes and densities, higher-fidelity representation of device environment
Bad: Unrealistically fast, fewer memory constraints, not case sensitive (unlike device) Slow, slow, did we say slow? Debugging tools have limited view of Ti objects
